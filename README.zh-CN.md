# Expense Assistant

[English](./README.md) | **中文**

Expense Assistant 是一个以本地优先、聊天交互为核心的报销准备工具，目标是显著降低报销整理过程中的手工劳动。

> 当前状态：`Planning`

## 项目概述

真正让人头疼的，往往不是花钱本身，而是报销准备过程。打车、吃饭、住宿这些费用发生以后，后面还要反复做很多机械工作：收集截图、整理发票、统一命名、回忆上下文、分类归档，最后再把内容一条条手工填进公司的报销系统。

Expense Assistant 的第一目标，就是先解决这一步。

在 v1 阶段，这个项目聚焦于把以下聊天式输入：

- 简短文字说明
- 语音消息
- 支付截图
- 纸质票据照片
- PDF 电子发票

整理成一份结构化的报销包，让用户可以快速核对，并方便地手工复制到 SAP 或 Concur 中。

这个项目的目标不是“一步全自动提交报销”，而是在保证可控、透明、安全的前提下，尽可能消灭重复性的报销准备工作。

## 为什么 v1 先做“报销准备”，而不是直接自动填 SAP

很多企业报销系统，尤其是基于 SAP 云环境或公司定制流程的系统，并不适合一开始就做自动提交。即使技术上可以尝试浏览器自动化，也常常会遇到以下问题：

- API 不明确，甚至没有开放 API
- 页面流程脆弱，改版后容易失效
- 登录、MFA、多重校验复杂
- 存在合规和公司政策风险
- 长期维护成本高

因此，v1 明确停在“自动整理，人工确认并提交”这一步。

Expense Assistant 不承诺一开始就打通 SAP，而是先把报销资料准备好、整理好、命名好、导出好，让用户可以更快地完成最后的人工录入。这是更现实、也更稳妥的切入方式。

## 核心工作流

计划中的 v1 工作流如下：

1. 用户通过本地网页聊天界面与助手对话。
2. 发送文本、语音、截图、票据照片或 PDF。
3. 助手自动提取报销所需的关键字段。
4. 如果某些信息不清楚，助手会在聊天中发起简短追问。
5. 用户快速复核整理后的费用条目。
6. 导出一份适合手工提交到 SAP 的报销包。

整个交互更像“给个人报销机器人发消息”，而不是传统表单式录入工具。

## 产品原则

### 低摩擦

输入方式应该尽量自然，像发消息一样使用，而不是要求用户先手工整理一遍再交给系统。

### 本地优先

在没有云端部署、没有企业系统集成之前，这个工具也应该可以直接在个人设备上发挥价值。这有助于降低使用门槛，也更利于控制敏感数据。

### 聊天优先

系统应以对话为主，而不是强制用户进入僵硬的流程。缺失的信息可以通过一句追问来补齐。

### 人在回路中

最终的报销结果必须可检查、可解释、可确认。系统负责整理、抽取、建议，但不应该在用户不知情的情况下向企业系统静默提交。

## v1 范围

v1 计划包含以下能力：

- 一个本地网页聊天界面，用于接收报销输入
- 支持文本、语音、图片、截图、PDF
- 对简短语音进行转写
- 对截图、票据、电子发票进行 OCR 和内容理解
- 提取结构化报销字段，包括：
  - 日期
  - 商户或供应商
  - 金额
  - 币种
  - 报销类别
  - 支付方式
  - 出差或项目上下文
  - 自由备注
- 当关键信息缺失或置信度较低时，在聊天里追问
- 维护会话级上下文，例如当前出差、项目、默认分类等
- 生成 Markdown 和 CSV 导出
- 统一附件命名，并建立附件索引

## v1 明确不做的内容

以下内容不属于第一版：

- 直接提交 SAP 或 Concur
- 把浏览器自动化作为默认工作流
- 企业审批流
- 多用户权限体系
- 财务系统集成
- 超出基本整理与标记范围的报销政策引擎
- 对非官方个人微信自动化的稳定支持承诺

项目方向会保留“类似微信机器人”的交互感，但 v1 会先围绕通用聊天接口设计，这样未来若接入企业微信、飞书或其他渠道，不需要重写核心整理逻辑。

## 规划中的架构

初始架构计划分为四层：

### 1. 聊天接入层

通过本地网页聊天界面统一接收消息和附件，并转换为标准输入格式。

### 2. 理解与抽取流水线

这一层负责：

- 语音转写
- OCR
- 票据解析
- 字段标准化
- 置信度评分
- 重复检测

### 3. 会话与追问层

这一层负责维护短期报销上下文，并在字段缺失或含糊时向用户发起追问。

### 4. 导出与归档层

这一层负责生成结构化导出文件，并对附件进行统一命名和归档。

## 导出结果

计划中的 v1 导出包包括：

- `expenses.md`：便于人工核对的报销摘要
- `expenses.csv`：适合复制、筛选、后续导入的结构化表格
- `attachments/index.csv`：附件重命名后的索引，以及附件与费用项的映射关系

这个导出设计的目标，就是让手工填 SAP 更快，而不是假设 SAP 已经可以自动接入。

## 路线图

### 里程碑 1

### 里程碑 2

完成本地聊天入口和基础文件上传能力。

### 里程碑 3

加入语音转写、OCR 和结构化报销字段抽取。

### 里程碑 4

加入追问机制、附件索引和导出能力。

### 里程碑 5

评估接入企业微信、飞书或其他受支持的消息渠道。

### 里程碑 6

只有在 v1 已经足够有用之后，才评估是否值得做 SAP 侧自动化，以及它是否合规、稳定、可维护。

## 隐私与合规说明

报销数据通常包含个人、财务以及雇主相关信息，因此项目从一开始就采取本地优先策略，同时避免轻易承诺可能触碰公司政策边界的系统自动化能力。

未来如果要接入企业聊天工具或报销系统，需要结合以下因素进行单独评估：

- 公司安全要求
- 数据保留策略
- 可接受使用政策
- 鉴权和登录约束
- 审计可追踪性
